# 敏捷与迭代开发

## 敏捷宣言

### 核心价值观

| 重视 | 超过 |
|------|------|
| 个体与互动 | 流程与工具 |
| 可工作的软件 | 详尽的文档 |
| 客户合作 | 合同谈判 |
| 响应变化 | 遵循计划 |

### 原则

1. 最高价值：尽早且持续交付有价值的软件
2. 欢迎变化：即使在后期也欢迎需求变化
3. 短周期交付：频繁交付可工作软件
4. 每日协作：业务与开发人员每天合作
5. 激励团队：营造信任和支持的环境
6. 面对面沟通：团队内部首选方式
7. 可工作软件：进度的主要度量
8. 可持续开发：保持稳定节奏
9. 追求卓越：持续关注技术设计
10. 简洁：减少不必要工作
11. 自组织团队：最好的架构来自自组织
12. 反思调整：团队定期反思并调整

## 敏捷方法

### Scrum

| 角色 | 职责 |
|------|------|
| **产品负责人 (PO)** | 需求优先级、价值最大化 |
| **Scrum Master** | 流程优化、障碍移除 |
| **开发团队** | 自我管理、交付成果 |

### 仪式

| 仪式 | 频率 | 时长 | 目的 |
|------|------|------|------|
| **Sprint 规划** | 每迭代 | 8h | 确定迭代目标和工作 |
| **每日站会** | 每天 | 15min | 同步进展、识别障碍 |
| **Sprint 评审** | 迭代末 | 4h | 演示成果、获取反馈 |
| **Sprint 回顾** | 迭代末 | 3h | 总结改进 |

### 工件

| 工件 | 说明 |
|------|------|
| **产品待办列表 (PBL)** | 所有需求的优先排序列表 |
| **Sprint 待办列表** | 当前迭代的需求和工作 |
| **产品增量** | Sprint 结束时交付的可工作软件 |

### Sprint 周期

```
规划 → 执行 → 评审 → 回顾 → 规划
```

## 看板 (Kanban)

### 核心原则

1. 可视化工作流
2. 限制在制品 (WIP)
3. 管理流动
4. 明确策略
5. 建立反馈环
6. 协作改进

### 看板面板

```
待办 → 进行中 → 测试 → 完成
```

## XP (极限编程)

| 实践 | 说明 |
|------|------|
| 结对编程 | 两人协作编码 |
| 持续集成 | 频繁合并代码 |
| 测试驱动 | 先写测试再写代码 |
| 重构 | 持续优化代码 |
| 简洁设计 | 避免过度设计 |

## 迭代 vs 增量 vs 敏捷

| 方法 | 特点 | 交付方式 |
|------|------|----------|
| **迭代** | 逐步完善 | 最后一次性交付 |
| **增量** | 分块交付 | 逐步交付各块 |
| **敏捷** | 迭代+增量 | 频繁交付可用软件 |

## 敏捷适用场景

| 适用 | 不适用 |
|------|--------|
| 需求变化多 | 需求非常稳定 |
| 需要快速响应 | 合同固定、变更困难 |
| 小型/中型团队 | 大型分布式团队 |
| 创新项目 | 监管严格领域 |
